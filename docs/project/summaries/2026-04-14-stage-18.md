# Stage 18 Summary

## Milestone

- Stage: 18
- Date: 2026-04-14
- Status: Completed

## Root Cause

- The wordbook tab was still truncating the visible cache to the first 12 entries, so the UI could never match the claimed wordbook size.
- Partial preprocess cache writes were storing synthetic placeholder entries as if they were finished results, so later runs could skip genuinely missing terms.
- Progress text reused the same count fields for different meanings, which made `0 / N` look like failure even when the system was still only selecting terms.
- Older placeholder-heavy caches from previous builds could still be loaded directly into the panel, preserving a false sense of completeness.

## What Was Done

- Removed the wordbook tab slice limit so the panel now renders the full current-file wordbook cache.
- Changed preprocess cache merging so only real generated entries are persisted during chunked writes.
- Ignored placeholder-like legacy entries when reusing cache state and when deciding which terms still need preprocessing.
- Added explicit progress semantics for candidate pool size, selected target count, cached entry count, and processed batches.
- Sanitized legacy placeholder wordbook caches when reopening a file so old incomplete caches are rewritten into a clean state before display.
- Added regression coverage for placeholder-cache reuse and verified the updated flow through compile and test.

## Outcome

The file wordbook pipeline is now internally consistent: the cache only treats real generated entries as completed work, older broken caches are cleaned up automatically, the wordbook tab shows the full available result set, and preprocess progress communicates the actual stage of work instead of ambiguous counts.
