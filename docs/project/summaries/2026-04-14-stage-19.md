# Stage 19 Summary

## Milestone

- Stage: 19
- Date: 2026-04-14
- Status: Completed

## Root Cause

- The preprocess panel still showed batch counters during the candidate-selection phase, so users could see misleading `0 / N` batch progress before any real preprocessing batch had started.
- Wordbook candidate extraction still missed some class-local methods, especially member-style references such as `self.xxx(...)`.
- The wordbook tab still rendered as one flat list, which made larger files hard to browse and did not reflect the actual class/function structure of the source file.

## What Was Done

- Stopped selection and preparation progress states from exposing batch counters before chunked preprocessing actually begins.
- Added member-function extraction for `self.xxx(...)`, `cls.xxx(...)`, and `this.xxx(...)` references.
- Added a scope analysis pass that infers class and function ranges from the active file and annotates visible wordbook entries with scope paths.
- Replaced the flat wordbook list with a compact collapsible tree grouped by classes, functions, and module scope.
- Added regression tests for member-function extraction, scope annotation, and selection-phase batch progress semantics.

## Validation

- `npm.cmd run compile`
- `npm.cmd test`
- Regression coverage confirmed:
  - Python member-function references enter the glossary and candidate pool
  - visible wordbook entries receive stable class/function scope paths
  - selection-phase progress snapshots keep `batchCount = 0` until actual preprocessing starts

## Outcome

The wordbook now behaves more like a navigable code dictionary: progress output no longer shows fake batches during selection, class-local member functions are more likely to appear in the candidate pool, and the wordbook page is organized into a compact collapsible scope tree that is much easier to scan in real files.
