# Stage 30 Summary

## Milestone

- Stage: 30
- Name: Multi-endpoint remote failover
- Date: 2026-04-15

## Goal

- Keep the original OpenAI-compatible endpoint as the primary remote path.
- Add a second remote endpoint as fallback.
- Make the extension try another remote endpoint before dropping all the way back to the local heuristic provider.
- Restore the user's local dev configuration without committing API secrets.

## What Changed

- Added `provider.fallbacks` configuration support.
- Extended `ExtensionSettings` with structured fallback endpoint definitions.
- Updated `getSettings()` so fallback endpoints can come from VS Code config or env.
- Updated the settings panel so fallback endpoints can be edited as line-based entries.
- Changed the OpenAI-compatible provider request path from single-endpoint retries to ordered endpoint failover.
- Added regression coverage for "primary endpoint fails, fallback endpoint succeeds".

## Endpoint Failover Behavior

- Request flow is now:
  - primary configured endpoint
  - fallback endpoint 1
  - fallback endpoint 2, if configured later
  - then local provider only if the higher-level caller already has local fallback logic
- Each endpoint keeps the existing payload retry pattern.
- Abort-like cancellation still stops immediately.
- Ordinary remote errors now move to the next configured endpoint instead of stalling on a single remote path.

## Local Environment Restore

- Restored local dev env in `.vscode/.env` so the primary endpoint is:
  - `https://airouter.service.itstudio.club/v1`
- Added the new fallback endpoint:
  - `https://f.sillydream.top/v1`
- Secrets remain local-only because `.vscode/` stays git-ignored.
- Workspace settings already pointed at the original primary base URL, so no committed settings change was required for secrets.
