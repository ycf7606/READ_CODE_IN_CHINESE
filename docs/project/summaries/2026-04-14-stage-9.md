# Stage 9 Summary

## Milestone

- Stage: 9
- Date: 2026-04-14
- Status: Completed

## Root Cause

- The extension read provider settings through `vscode.workspace.getConfiguration(...).get(...)`.
- Because contributed settings in `package.json` already define defaults such as `provider.id = local`, the schema defaults were returned even when the active workspace had no explicit provider settings.
- During Extension Development Host testing against another workspace, the repo-local `.vscode/settings.json` did not apply to that target workspace, so the extension silently stayed on the local heuristic provider.

## What Was Done

- Reworked settings resolution to prefer explicit VS Code configuration and otherwise fall back to environment defaults.
- Expanded the local ignored `.vscode/.env` file so the development host carries provider id, base URL, model, and hyperparameters across workspaces.
- Added clearer diagnostics for effective settings and provider selection.
- Surfaced engine source and fallback notes directly in the explanation panel.
- Repaired corrupted follow-up suggestion text in both local and remote paths.

## Outcome

The development host can now use the real remote provider even when testing the extension against a different workspace, as long as the local `.vscode/.env` contains the provider defaults. When the extension still falls back to local analysis, the panel and logs now make that state explicit instead of looking like a successful remote analysis.
