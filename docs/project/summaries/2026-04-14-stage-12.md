# Stage 12 Summary

## Milestone

- Stage: 12
- Date: 2026-04-14
- Status: Completed

## Root Cause

- Stage 11 added a broad token-prebuild workflow, but the live product still had two practical gaps: stale requests could keep running after the user had already changed context, and single-token prompts still lacked enough exact callsite detail to avoid generic answers.
- The old prebuild direction also did too much work at the language level when the real user need was faster, file-specific explanations for the symbols they were actually reading right now.

## What Was Done

- Replaced the broad token-prebuild-first path with a file-scoped preprocess cache for user-defined variables, functions, classes, and types.
- Added a batch preprocess builder that ranks current-file glossary symbols, sends one full-file request to the remote provider, and stores short symbol summaries in `.read-code-in-chinese/preprocess/`.
- Changed token explanation flow so it now checks the current file's preprocess cache first, then the older token cache, before issuing a fresh remote explanation.
- Added explicit stale-task cancellation for explanation, follow-up, and preprocess work so the newest selection or editor change wins.
- Improved token prompts with exact selection-line previews and glossary hints, and hardened remote response parsing against alternate payload content shapes.
- Hardened settings persistence by routing reasoning-effort updates through the same persistence path, sanitizing numeric values, and scoping preprocess progress to the active file only.
- Updated README, architecture notes, workboard, changelog, and project logs to reflect the new preprocess-first behavior.

## Outcome

The extension now behaves much closer to the intended product: repeated single-symbol explanations can come back quickly from a file-scoped cache, stale work is canceled instead of queued behind old selections, and single-token remote explanations are grounded in the exact callsite rather than drifting into template language.
